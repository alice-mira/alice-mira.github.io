<p class="line-2" style="margin-block: 16px; font-size: 1rem;">在 Linux 系统中，资源限制管理是确保系统稳定性、安全性和公平性的核心机制。无论是防止单个用户进程耗尽所有内存导致系统崩溃，还是为容器化应用分配可控的 CPU、内存配额，亦或是限制后台任务对磁盘 I/O 的过度占用，资源限制都扮演着至关重要的角色。与 Windows 或 macOS 的图形化资源管理器不同，Linux 提供了从传统 Shell 级别限制（ulimit）到现代内核级控制组（cgroups）的多层次、精细化管理工具。本文将系统性地介绍 Linux 中资源限制的各类方法、适用场景、配置步骤以及最佳实践，帮助你构建一个健壮且可控的系统环境。</p><h2 id=1.-为什么需要资源限制？ class="line-4" style="margin-block: 16px; font-size: 1.5rem; font-weight: bold; color: oklch(72.3% 0.219 149.579); border-left: 4px solid oklch(72.3% 0.219 149.579); border-bottom: 1px solid oklch(72.3% 0.219 149.579); padding-left: 4px;">1. 为什么需要资源限制？</h2><p class="line-6" style="margin-block: 16px; font-size: 1rem;">在深入命令之前，理解资源限制的必要性至关重要。</p><h3 id=1.1-防止资源耗尽 class="line-8" style="margin-block: 16px; font-size: 1.375rem; font-weight: bold; color: oklch(72.3% 0.219 149.579); border-color: oklch(72.3% 0.219 149.579);">1.1 防止资源耗尽</h3><ul class="line-9" style="margin-block: 0px;"><li class="line-9" style=""><strong class="line-9" style="font-weight: bold;">内存耗尽（OOM）</strong>：一个内存泄漏的进程可能吃光所有物理内存和交换空间，触发内核的 OOM Killer 随机终止进程，可能导致关键服务意外死亡。</li><li class="line-10" style=""><strong class="line-10" style="font-weight: bold;">CPU 占用 100%</strong>：一个陷入死循环或设计不良的进程可能独占一个或多个 CPU 核心，导致系统响应迟缓，其他任务无法获得计算资源。</li><li class="line-11" style=""><strong class="line-11" style="font-weight: bold;">磁盘空间爆满</strong>：日志文件无限增长或失控的文件写入可能填满磁盘，导致系统无法创建新文件、写入日志，甚至无法启动新进程。</li><li class="line-12" style=""><strong class="line-12" style="font-weight: bold;">文件描述符耗尽</strong>：未关闭网络连接或文件句柄的进程可能耗尽系统允许的最大文件描述符数，导致无法建立新的网络连接或打开文件。</li></ul><h3 id=1.2-实现资源隔离与公平性 class="line-14" style="margin-block: 16px; font-size: 1.375rem; font-weight: bold; color: oklch(72.3% 0.219 149.579); border-color: oklch(72.3% 0.219 149.579);">1.2 实现资源隔离与公平性</h3><ul class="line-15" style="margin-block: 0px;"><li class="line-15" style=""><strong class="line-15" style="font-weight: bold;">多用户系统</strong>：在服务器上，需要防止一个用户的失控进程影响其他用户的正常使用。</li><li class="line-16" style=""><strong class="line-16" style="font-weight: bold;">容器与虚拟化</strong>：Docker、Kubernetes、LXC 等容器技术依赖 cgroups 为每个容器分配独立的资源配额，实现轻量级隔离。</li><li class="line-17" style=""><strong class="line-17" style="font-weight: bold;">服务质量（QoS）</strong>：确保高优先级服务（如数据库）能获得足够的资源，而低优先级任务（如批量数据处理）不会干扰前者。</li></ul><h3 id=1.3-安全加固 class="line-19" style="margin-block: 16px; font-size: 1.375rem; font-weight: bold; color: oklch(72.3% 0.219 149.579); border-color: oklch(72.3% 0.219 149.579);">1.3 安全加固</h3><p class="line-20" style="margin-block: 16px; font-size: 1rem;">限制进程可用的资源也是安全最佳实践的一部分，可以减缓某些拒绝服务（DoS）攻击或限制被入侵进程的破坏范围。</p><h2 id=2.-传统-Shell-级限制：ulimit class="line-22" style="margin-block: 16px; font-size: 1.5rem; font-weight: bold; color: oklch(72.3% 0.219 149.579); border-left: 4px solid oklch(72.3% 0.219 149.579); border-bottom: 1px solid oklch(72.3% 0.219 149.579); padding-left: 4px;">2. 传统 Shell 级限制：ulimit</h2><p class="line-24" style="margin-block: 16px;
